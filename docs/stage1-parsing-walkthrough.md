# Parsing walkthrough

The parser in `src/phish_triage/parser.py` reads one `.eml` file and returns a `ParsedEmail` dataclass. The module uses the Python standard library and performs no network access.

This walkthrough uses `tests/fixtures/phish_sample.eml`, an inert message generated by `tests/make_fixtures.py`.

## Processing flow

```text
.eml file
  -> email.message_from_binary_file(..., policy=email.policy.default)
  -> header and address parsing
  -> Received-chain parsing
  -> MIME text and attachment extraction
  -> named signal derivation
  -> JSON or Markdown rendering
```

## Authentication results

`parse_auth_results()` examines every `Authentication-Results` header for SPF, DKIM, and DMARC result tokens. Supported values are:

```text
pass, fail, none, neutral, softfail, temperror, permerror, policy, bestguesspass
```

Each mechanism starts as `none`. Once a non-`none` result has been recorded, a later header does not replace it. This is a small precedence rule, not a complete implementation of the Authentication-Results grammar.

The phishing fixture produces `spf=fail`, `dkim=none`, and `dmarc=fail`, which derive the `spf_fail` and `dmarc_fail` signals.

## Sender checks

`parse_address_mismatch()` uses `email.utils.parseaddr()` to split the `From` and `Reply-To` headers.

It derives `from_reply_to_mismatch` when both addresses exist and their complete address strings differ. The implementation compares full addresses, not only their domains.

It derives `display_name_spoof` in either of these cases:

- the display name contains an email address different from the actual From address; or
- the display name contains one of the configured brand terms and that same term does not appear anywhere in the From address domain.

The configured terms are `paypal`, `microsoft`, `apple`, `google`, `amazon`, `irs`, `bank`, `chase`, `wells fargo`, `dhl`, and `fedex`. This is a heuristic substring check, not domain-ownership validation.

## Received chain

`parse_received_chain()` reverses the message's `Received` headers so that index 0 represents the earliest parsed hop. For each header it attempts to extract:

- the sending host and IP address;
- the receiving host;
- a UTC ISO 8601 timestamp; and
- whether the sending IP is internal.

Internal addresses include RFC 1918 IPv4 space, IPv6 unique-local addresses, loopback addresses, and link-local addresses. Documentation ranges such as `198.51.100.0/24` are intentionally treated as public so they can stand in for public hosts in the fixtures.

`detect_private_to_public_anomaly()` derives `received_private_to_public` when an internal hop is followed by a hop with a parsed non-internal IP address.

Received-header syntax varies widely. The parser uses permissive regular expressions and should be treated as an aid to analysis, not an authoritative reconstruction of message transport.

## URLs

`extract_urls()` walks every non-attachment `text/*` MIME part, decodes its transfer encoding and character set, and extracts unique `http://` and `https://` strings in encounter order.

`defang()` then replaces `http` with `hxxp` and every period with `[.]`. The original URLs remain in the structured `urls` field for enrichment, while `urls_defanged` is used in analyst output.

Two URL heuristics can add signals:

- `url_shortener` for `bit.ly`, `tinyurl.com`, `t.co`, `goo.gl`, `ow.ly`, or `is.gd` substrings;
- `suspicious_tld` for URLs matching `.zip`, `.mov`, `.top`, `.xyz`, `.click`, `.country`, or `.support` near the end of the path before a query string.

These are substring and suffix heuristics rather than Public Suffix List parsing.

## Attachments

`extract_attachments()` includes a non-multipart MIME part when it has an attachment disposition or a filename. For each result it records the decoded filename, content type, byte length, MD5, SHA-1, and SHA-256. The presence of at least one attachment derives `has_attachment`.

The parser computes hashes locally and does not open or execute attachment content.

## Output schema and rendering

`ParsedEmail.to_dict()` converts the dataclass tree into JSON-serializable dictionaries and lists. The CLI provides two parser output forms:

```bash
phish-triage parse tests/fixtures/phish_sample.eml --json
phish-triage parse tests/fixtures/phish_sample.eml --markdown
```

Use `--output PATH` with either format to write the result to a file. JSON is the default when neither `--json` nor `--markdown` is supplied.

The phishing fixture currently derives these eight signals:

```text
spf_fail
dmarc_fail
from_reply_to_mismatch
display_name_spoof
received_private_to_public
url_shortener
suspicious_tld
has_attachment
```

`dkim_fail` is a ninth supported signal but does not appear in that fixture because its DKIM result is `none`.
